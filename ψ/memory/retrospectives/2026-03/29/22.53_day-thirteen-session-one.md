## Honest Feedback

**1. Pile-on norm is toothless.** Despite being in every /recap, despite being called out twice in this session, Beasts still pile on. The norm says "SHOULD" — maybe it needs to be a MUST. Or maybe the tooling needs to change: if 2+ Beasts have already responded to a message, new responses should get a warning. Culture alone is not enough.

**2. CF cache will keep biting us.** The content-length fix helps, but Gorn still needs to manually purge CF cache to see frontend changes. This is a terrible UX. We need cache-busting in the build pipeline — hashed filenames, proper Cache-Control headers, or automatic CF purge on deploy. Every session has at least one "hard refresh" frustration.

**3. Leonard as message relay is noisy.** I spent most of this session acknowledging Gorn's DMs, posting to forum, and notifying him when fixes shipped. That is necessary but it fills thread #374 with 100+ messages. The signal-to-noise ratio drops. Maybe Sable should handle more of the relay work, or the DM→forum→DM loop should be automated.

## Lessons Learned

1. **Content-length headers matter for CDN caching** — Hono's c.body() with a Buffer doesn't set content-length automatically. CDNs trust the header over the body.
2. **Pile-on norms need enforcement mechanisms, not just cultural pressure** — calling it out works once, then behavior reverts.
3. **The Secretary pattern works** — routing Gorn's requests through a structured pipeline (DM → forum → Zaghnal → builder → reviewer → done) kept 35+ tasks flowing without collisions.
