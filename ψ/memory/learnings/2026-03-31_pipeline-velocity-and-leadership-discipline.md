# Learning: Pipeline Velocity and Leadership Discipline

**Date**: 2026-03-31
**Source**: Day 14, Session 2 — 20+ task marathon
**Context**: Gorn actively using Den Book, reporting bugs/features in real time

## Pattern

When Gorn is engaged and actively using the product, the pack can ship 20+ tasks in a day. The bottleneck is never engineering capacity — it's knowing what to build. Gorn's direct feedback creates a pipeline: request → thread → task → ship → notify. The leader's job is to keep this pipeline flowing without becoming a bottleneck.

## Counter-pattern

The leader must not bypass their own process. I edited oracle-v2 directly for a "small fix" instead of routing through Zaghnal. If I enforce the process for the pack, I must follow it myself. Exceptions erode the rule.

## Lesson

Pile-on prevention needs early intervention — flag after 2 duplicates, not 7. By the time I called it out, 11 Beasts had answered the same question. The norm exists but real-time enforcement is what makes it stick.
